# dbs-hook R3 — 开头优化引擎

> 原始文档: `references/original_body.md` (5.7KB)

## 核心信条

| # | 信条 | 含义 |
|---|------|------|
| 1 | 开头是内容的试用装 | 必须独立工作，不能假设用户看了封面/标题 |
| 2 | 好开头 = 话题 + Hook + 可信度 | "我去年涨粉200万(Hook)，靠的就是搞清楚选题和标题的区别(话题)" |
| 3 | 开头制造悬念，不直接给答案 | ❌"李亚鹏30年干黄十几个项目，证明人脉≠赚钱" ✅"善良的李亚鹏认识半个娱乐圈，30年为什么赚不到钱" |
| 4 | 开头必须口播友好 | ❌"你以为你在自律？其实你只是在逃避执行" ✅"只要你没在执行，你的动作大概率就是为了逃避执行而生的动作" |

## 5 Phase 工作流

### Phase 1: 接收文案
确认用户提供了什么：完整文案(有开头+正文) / 只有正文(没开头) / 只有选题(连正文都没有)

### Phase 2: 诊断内容质量 (关键！)
**在优化开头之前，先诊断内容本身有没有问题。**
- 内容本身无价值 → 开头再优化也没用
- 内容缺少素材/冲击力 → 先补内容，再优化开头
- 内容有价值但表达不好 → 进入 Phase 3

### Phase 3: 3 种优化方法

| 方法 | 适用场景 | 操作 |
|------|---------|------|
| **素材提取** | 正文有强力素材但开头平淡 | 从正文提取最有冲击力的数据/案例/观点前移到开头 |
| **增补** | 内容好但开头缺钩子 | 基于话题补充 Hook + 可信度元素 |
| **悬念制造** | 开头直接给了答案 | 把答案改成问题/悬念/反常识陈述 |

### Phase 4: 口播检查
- 念一遍优化后的开头
- 标记拗口处 → 改成直接陈述
- 标记自问自答处 → 改成直接陈述

### Phase 5: 输出对比
```
❌ 原开头: [原文]
✅ 优化后: [新开头]
📐 方法: [素材提取/增补/悬念制造]
🎯 Hook: [具体钩子是什么]
```

## 失败模式与兜底

| 失败模式 | 原因 | 兜底 |
|---------|------|------|
| 内容本身无价值，开头再优化也没用 | 跳过 Phase 2 | 回到 Phase 2 先诊断内容质量 |
| 优化后开头仍然不留人 | 未制造悬念或开头=答案 | 检查是否直接给了答案→改为悬念 |
| 优化后不适合口播 | 未做 Phase 4 口播检查 | 念一遍标记拗口处重写 |
